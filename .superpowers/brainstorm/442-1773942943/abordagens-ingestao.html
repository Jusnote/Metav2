Sistema de Ingestão de Leis — 3 Abordagens

Analise cada abordagem e selecione a que mais faz sentido para o seu fluxo de estudo

A

Fix & Scale — Corrigir o Parser + Automatizar Planalto

Manter o pipeline atual, mas corrigir as lacunas do regex e automatizar o scraping do HTML do Planalto.

Pipeline:
Planalto HTML → Scraper automático → Parser regex melhorado → Plate.js JSON → Supabase

Prós

  • Mudança mínima na arquitetura
  • Parser já existe, só precisa expandir
  • HTML do Planalto tem tags <a> nas anotações — mais fácil de separar que texto puro
  • Implementação rápida (~1-2 semanas)

Contras

  • Regex nunca será 100% — sempre pode escapar um padrão novo
  • Planalto pode mudar o HTML a qualquer momento
  • Sem fonte oficial de verdade para validar
  • Escalar para 200 leis ainda depende de scraping frágil
B

LexML/Senado API — Fonte Estruturada Oficial

Usar as APIs governamentais (LexML XML + Senado Dados Abertos) como fonte primária. Anotações já vêm separadas na estrutura XML.

Pipeline:
LexML URN → API XML/JSON → XML Parser (estruturado) → Plate.js JSON → Supabase

Fallback:
Planalto HTML → Parser regex (Abord. A) → Supabase

Prós

  • Dados estruturados oficiais — confiáveis e estáveis
  • Anotações marcadas semanticamente no XML
  • URN persistente (urn:lex:br:federal:lei:...)
  • Automatizável para 200+ leis por batch
  • Possibilidade de atualização automática

Contras

  • API pode não ter texto integral de todas as leis
  • Precisa de fallback para leis não disponíveis
  • XML parsing mais complexo que regex
  • Confiabilidade da API (uptime) desconhecida
C

Multi-Source + Docling — Pipeline Robusto Universal

Pipeline com múltiplas fontes: LexML (primário), Planalto HTML (secundário), Docling para PDFs. Normaliza tudo para o mesmo formato.

Fontes:
① LexML API  |  ② Planalto HTML  |  ③ PDF (Docling)  |  ④ JusBrasil*

             ↓ normalização ↓

Formato Unificado → Separador de Camadas → Supabase

* JusBrasil: dados bem estruturados por dispositivo, mas requer cuidado legal com scraping

Prós

  • Aceita QUALQUER fonte — não depende de uma só
  • Docling resolve PDFs (DOU, compilações do Senado)
  • Formato normalizado único para todas as fontes
  • Máxima cobertura para as 200 leis
  • Camada de anotações separada by design

Contras

  • Mais complexo de implementar e manter
  • Docling é pesado (Python + modelo ML)
  • JusBrasil tem implicações legais de scraping
  • Over-engineering se as APIs Gov já cobrirem tudo

💡 Minha Recomendação: Abordagem B + elementos de A

LexML como fonte primária (dados já estruturados, anotações separadas no XML) com Planalto HTML como fallback (parser corrigido). Docling é overkill para este caso — o problema não é formato de documento (PDF), é separação de anotações, e isso se resolve melhor na fonte.

Sobre JusBrasil: Boa observação sua — eles realmente estruturam cada dispositivo como entidade. Mas scraping deles tem risco legal (LGPD + termos de uso). O LexML oferece a mesma estruturação sem esse risco, pois é dado aberto oficial.

Sobre Docling: Não recomendo como peça central. Docling brilha para converter PDFs/DOCX desorganizados → structured output. Mas as leis brasileiras já existem em formato estruturado (HTML/XML) nas fontes oficiais. Usar Docling seria adicionar complexidade (Python + modelo ML de 258M) para resolver um problema que se resolve melhor na camada de fonte de dados.
