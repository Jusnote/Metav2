Proposta: Pipeline de Ingestão de Leis com Qualidade Garantida

Arquitetura em 3 camadas: Ingestão Inteligente → Separação de Anotações → Validação Automática

Camada 1 — Ingestão Inteligente

O objetivo é extrair o máximo de estrutura possível da fonte, antes do parsing.

❌ HOJE (copy/paste)
Planalto.gov.br
→ Ctrl+C (perde HTML)
→ Texto puro no editor
→ Regex tenta adivinhar
→ Anotações escapam
✅ PROPOSTO (HTML direto)
Planalto.gov.br
→ Fetch HTML direto (URL)
→ Extrair <a> tags (anotações)
→ Parser HTML adaptativo
→ Anotações já separadas
O insight-chave: HTML variável ≠ anotações variáveis
Sim, cada lei tem HTML diferente. MAS as anotações legislativas seguem um padrão mais consistente mesmo entre leis: elas são quase sempre <a href="...">(Redação dada...)</a> ou texto entre parênteses com palavras-chave conhecidas. O parser HTML não precisa entender a estrutura inteira da lei — só precisa identificar e extrair as anotações antes de parsear o texto.

Estratégia: Primeiro remove todas as tags <a> que são anotações do HTML → salva separadamente → depois converte o HTML limpo pra texto → aí sim aplica o parser regex (que agora não precisa mais adivinhar anotações).

Camada 2 — Separação de Anotações em Camadas

Texto limpo para estudo + histórico legislativo sob demanda.

Camada 1: Lei Seca
Texto puro da lei, sem NENHUMA anotação. É o que o usuário vê por padrão durante o estudo.
Art. 121. Matar alguém:
Pena - reclusão, de seis a vinte anos.
Camada 2: Histórico
Anotações legislativas organizadas. Visível via hover/click/toggle no dispositivo.
⟡ Redação dada pela Lei nº 13.968/2019
⟡ Vigência
Camada 3: Minhas Notas
Anotações pessoais do usuário (caderno temático). Já existe no sistema.
📝 Homicídio simples é crime doloso...
Storage: Cada dispositivo no Supabase terá campos separados: texto_limpo (camada 1), anotacoes_legislativas[] (camada 2), texto_original (backup integral). A camada 3 já está no caderno_items.

Camada 3 — Validação Automática (Cinto e Suspensório)

Garantir que nenhum dispositivo fique faltando, truncado ou com anotação grudada.

🔢 Validação Estrutural
  • Contar artigos parseados vs. total esperado
  • Verificar sequência numérica (Art. 1, 2, ... N sem gaps)
  • Detectar artigos com texto vazio ou suspeitamente curto
  • Validar hierarquia (todo artigo tem path)
🔍 Validação de Anotações
  • Regex de segurança no texto_limpo buscando padrões residuais
  • Detectar parênteses com palavras-chave legislativas
  • Flaggar dispositivos onde texto_limpo ≈ texto_original (anotação não extraída?)
  • Score de confiança por dispositivo
🔄 Validação Cruzada
  • Comparar contagem de dispositivos entre Planalto e segunda fonte
  • Verificar texto de amostra aleatória (10% dos artigos) contra fonte 2
  • Detectar divergências de vigência (revogado numa fonte, ativo na outra)
📊 Relatório de Qualidade
  • Dashboard pós-ingestão: verde/amarelo/vermelho por lei
  • Lista de dispositivos flagrados para revisão manual
  • % de cobertura e score geral de confiança
  • Só 200 leis — revisão manual é viável nos flagrados
O princípio: Revisão manual só nos flagrados
Se a validação automática flagga 5% dos dispositivos, em 2.082 artigos do CC isso são ~104 artigos para revisar manualmente — viável. E se a ingestão for boa, pode ser 1% ou menos. O olho humano só precisa ver o que a máquina não tem certeza.

Resumo da Proposta

1. Ingestão: Fetch do HTML do Planalto (não copy/paste) → extrair anotações das tags HTML ANTES de parsear → parser regex no texto já limpo
2. Storage: texto_limpo + anotacoes_legislativas[] + texto_original separados por dispositivo
3. Validação: Contagem estrutural + regex de segurança no texto limpo + validação cruzada + relatório
4. UX: Lei seca limpa por padrão, indicador visual de histórico legislativo, popup/hover para ver anotações
5. Escala: Pipeline automatizável — informar URL do Planalto → ingestão + validação → relatório → revisão manual só dos flagrados
