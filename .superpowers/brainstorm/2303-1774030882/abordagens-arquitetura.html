Arquitetura do Sistema de Lei Seca — 3 Abordagens

Todas compartilham: API GraphQL no Hetzner, React puro + getSelection, Supabase para user data

A

Monolito GraphQL

Tudo num serviço Node.js: API GraphQL + script de extração + processamento + full-text search via PostgreSQL tsvector

Pros

  • Menor complexidade operacional — 1 container no Coolify
  • Deploy simples, debug simples
  • PostgreSQL tsvector é suficiente para busca textual
  • Mais rápido de implementar (~2-3 semanas)

Cons

  • Sem busca semântica (só textual exata)
  • Extração roda no mesmo processo da API
  • Se precisar escalar, tudo escala junto

Stack: Fastify + Mercurius + PostgreSQL + tsvector

B

API + Typesense (Recomendado)

API GraphQL + Typesense para busca: PostgreSQL é source of truth, Typesense sincronizado para busca rápida com facets. Mesmo pattern da API de questões.

Pros

  • Busca instant — typo tolerance, facets por tipo/nível/data
  • Mesmo pattern da API de questões (infraestrutura conhecida)
  • Typesense já roda no servidor — custo zero
  • Futuro: adicionar Voyage embeddings = busca semântica
  • Separação clara: PostgreSQL = CRUD, Typesense = busca

Cons

  • Sync PostgreSQL → Typesense (complexidade moderada)
  • 2 datastores para manter
  • ~1 semana a mais que opção A

Stack: Fastify + Mercurius + PostgreSQL + Typesense (existente)

C

Microserviços Separados

3 serviços independentes: API GraphQL, Worker de extração/processamento, Serviço de busca (Typesense). Comunicam via fila (Redis/BullMQ).

Pros

  • Isolamento total — extração pesada não afeta API
  • Cada serviço escala independente
  • Worker pode rodar 24/7 sincronizando leis novas
  • Arquitetura "enterprise-ready"

Cons

  • 3 containers no Coolify, 3 deploys, 3 logs
  • Complexidade desnecessária para o volume atual
  • Redis como message broker adiciona ponto de falha
  • ~3-4 semanas a mais que opção A

Stack: Fastify + Mercurius + PostgreSQL + Typesense + Redis/BullMQ

Minha Recomendação: B

A opção B é o sweet spot perfeito. Reutiliza infraestrutura que já existe (Typesense, pattern da API de questões), sem overengineering de microserviços. A extração roda como script CLI separado (não precisa ser serviço), o processamento roda uma vez na ingestão, e o sync para Typesense é um job simples.

O caminho para busca semântica futura (Voyage-4-large) já fica pavimentado — basta adicionar embeddings ao Typesense index, exatamente como funciona na API de questões.
